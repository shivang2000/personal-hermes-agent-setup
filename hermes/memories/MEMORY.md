EC2 deploy pitfalls (learned 2026-07-12): (1) `mt5_data` bind-mount dir must be chown 911:911 BEFORE first start — otherwise Wine says "is not owned by you" and curl downloads fail (`sudo chown -R 911:911 mt5_data` from EC2-user shell works; ec2-user is in wheel+sudo NOPASSWD). (2) Box has `docker-compose` v5.0.1 standalone (not `docker compose` plugin) and buildx 0.12 — incompatible. Workaround: `docker build -t trading-bot-v2-trading-bot:latest .` directly, THEN `docker-compose -f docker-compose.ec2.yml up -d`. (3) IMDSv1 is disabled (returns 401), need IMDSv2 token. (4) Bot crash-loops on "MT5 result expired" when MT5 has no logged-in account — log in via noVNC FIRST, then `docker-compose -f docker-compose.ec2.yml restart trading-bot` (the restart forces `mt5.initialize()` to re-attach to the now-logged-in terminal). (5) After the user logs in via noVNC, the RPyC server's Wine-Python `mt5.initialize()` may already be in a stale state; bouncing ONLY the bot container fixes it. (6) main.py fatal-exits when both MT5 and Telegram are down. Telegram isn't configured for this deploy (SignalGenerator strategies are self-contained), so Telegram-skip is expected — bot stays "DEGRADED" but runs. (7) Bot's DiscordNotifier alerts go to the PARENT #trading-bot channel (1525476646176690297), NOT the thread — operational log vs discussion. (8) Demo broker `FundingPips-SIM1` doesn't carry US30 symbol — bot caches 9/10 instruments and continues. (9) For live funded account, the noVNC login step is fragile; TODO: modify src/mt5/client.py:79 to pass MT5_LOGIN/MT5_PASSWORD/MT5_SERVER env vars to mt5.initialize() so the bot auto-logs in.
